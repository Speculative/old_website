Terminology

  • Cryptology - The study of codes and ciphers
  • Encryption - A process that reversibly converts plaintext to ciphertext
  • Code - A means of replacing words or phrases of plaintext with code words
  • Cipher - A cipher "enciphers" plaintext into ciphertext
  • Plaintext - Ordinary, human-readable text
  • Ciphertext - Unreadable text that is generated by encrypting plaintext and can be decrypted back into plaintext
  • Key - A "shared secret" between the two parties trying to communicate secretly that allows one party to decrypt the other's encrypted message
  • Cryptanalysis - The study of breaking ciphers so we can get plaintext from just the ciphertext without having access to the key

A Brief Math Detour

Modular Arithmetic

  • Based around integer division
  • Notated as a ≡ b (mod n) or a mod n = b
  • We say that a number is "congruent modulo n" if n/a = (something) + b
  • In other words, b is the remainder after we perform integer division of n by a
  • We can also think of the numbers "wrapping around" to 0 when we count up to n, e.g. 5 mod 3 we would count 1, 2, 3, 4, 5 as 1, 2, 0, 1, 2
  • You can intuitively confirm this with some quick mental math:
    5 = 3 · 1 + 2, thus 5 ≡ 2 (mod 3)

Caesar Cipher

  • One of the most basic kinds of ciphers
  • Consists of an alphabetic "shift" of all of the letters in plaintext into the letters in ciphertext that "wraps around" the letter A
  • We can think of this like being modular arithmetic. If we assign numbers 0 to 25 (26 in total) to the letters A-Z (i.e. A = 0, B = 1, ...) then a Caesar Shift of a plaintext letter whose number is n enciphers to ciphertext letter n + shift mod 26
  • For example, a shift of the letter C by 3 gives us F (C = 3; 3 + 3 = 6; F = 6)
  • An example of wraparound: Shift X by 7 (X = 23; 23 + 7 mod 26 = 4; E = 4)
  • We also refer to a shift of a certain number by the letter with that corresponding number (e.g. a Caesar Shift of 5 is called a Caesar F Shift)

Caesar Cipher - Encrypting

  • APPLE in a Caesar D Cipher:
  • A (0) + D (3) = 3 = D
  • P (15) + D (3) = 18 = S
  • L (11) + D (3) = 14 = O
  • E (4) + D (3) = 7 = H
  • APPLE encrypts to DSSOH in a Caesar D cipher

Caesar Cipher - Decrypting

  • We want to invert the encryption operation to decrypt ciphertext to plaintext
  • To reverse the shift, we shift backwards instead of forwards
  • So to reverse our Caesar D Cipher with ciphertext DSSOH:
  • D (3) - D (3) = 3 = A
  • S (18) - D (3) = 15 = P
  • O (14) - D (3) = 11 = L
  • H (7) - D (3) = 4 = E

Caesar Shift - Cryptanalysis

  • We don't really try to perform cryptanalysis on the Caesar Cipher because it's so simple.
  • What we might consider an approach to cryptanalysis is called brute force
  • Brute Force analysis is analysis through exhaustion of all possible inputs
  • Brute force is easy in this case because we have only 26 possible letters to try (25 if we discount the useless Caesar A shift)
  • So we could break a Caesar-enciphered ciphertext by just trying every possible shift until we got something that looked correct

Monoalphabetic Ciphers

  • We can think of this mapping as a "dictionary" of sorts
  • To encipher, we would look up a plaintext letter and find the corresponding ciphertext letter
  • To decipher, we look up a ciphertext letter and find the corresponding plaintext
  • Unlike the Caesar Cipher, enciphering works through this lookup operation, rather than "shifting"

Frequency Analysis

  • We can abuse the fact that certain letters in the English language appear more often than others
  • For example, E is the most common letter in the English language
  • Since the same plaintext letter always enciphers to the same ciphertext letter, the letter that occurs most frequently in the plaintext probably corresponds to the plaintext letter E

Vigenère Cipher

  • Polyalphabetic cipher
  • Instead of a lookup table, we have a secret word as our key
  • We tile this word across our plaintext and, for each letter in the plaintext, we encipher by Caesar shifting according to the corresponding letter of the key word
Key:         CLIMBCLIMBCLIMBC
Plaintext:   ENCRYPTIONISEASY
Ciphertext:  GYKDZREQAOKDMMTA

Vigenère Cipher - Cryptanalysis

  • Decryption is easy for the person who has the key. But we don't have the key!
  • We can't perform frequency analysis, because the "polyalphabetic" nature of the cipher makes it so that relative frequencies don't work
  • The same plaintext letter will encrypt to different ciphertext letters depending on which letter of the key corresponds to it at that position
  • Clearly we're going to have to figure out the key
  • Then we can just decrypt the message!

Babbage-Kasiski Analysis

  • If the key is relatively short compared to the length of the text, we can perform cryptanalysis
  • First, we want to find out how long the key is
  • We find repeated digraphs or trigraphs (sequences of 2-3 characters)
  • Then, by looking at the distances between those sequences, we can deduce the length of the key

Babbage-Kasiski Analysis

  • We note that every letter corresponding to the same letter in the keyword has the same caesar shift
  • This means we can perform frequency analysis!
  • So we perform frequency analysis and Caesar decryption on each letter of the keyword, giving us our key, and allowing us to decrypt the ciphertext

One Time Pads

  • Notice that this method works by finding repeated sequences of characters
  • This also depends on the keyword being replicated many many times across the ciphertext
  • So to perform successful analysis, we need a key that we can perform frequency analysis against, as well as a key that's shorter than the length of the text
  • Thus, if we had a key that was as long as the text we wanted to encrypt, and the key was randomly generated, we wouldn't be able to perform Babbage-Kasiski Analysis
  • It's actually been proven (there's an entire academic field called "Information Theory" founded by Claude Shannon) that One Time Pads with perfectly random keys that are as long as the enciphered text provide perfect secrecy
  • Drawbacks: distributing keys like this is extremely impractical, and most attempts at this in the real world have had holes of some kind
